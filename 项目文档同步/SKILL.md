# 项目文档同步

自动更新项目根目录下的6个核心文档，保持文档与代码同步。

---

## 管理的文档

| 文件名 | 中文名称 | 更新时机 |
|--------|----------|----------|
| 01_README.md | 项目说明 | 技术栈变化、功能增减、部署方式变更 |
| 02_项目地图.md | 项目地图 | 文件创建、删除、重命名、拆分 |
| 03_开发协议.md | 开发协议 | 新增规矩、规范变更 |
| 04_项目运作流程.md | 项目运作流程 | 核心流程变化、新增页面、API变更 |
| 05_审计违规清单.md | 审计违规清单 | 发现违规、处理完成、审计总结 |
| 06_项目规划.txt | 项目规划 | 功能完成、新增计划、版本里程碑 |

---

## 触发条件

**必须调用此skill的情况：**

1. **文件操作后**
   - 创建新文件 → 更新项目地图
   - 删除文件 → 更新项目地图，必要时归档记录
   - 重命名文件 → 更新项目地图和所有引用
   - 拆分文件 → 更新项目地图，记录拆分信息

2. **功能变更后**
   - 新增功能 → 更新README、项目规划
   - 删除功能 → 更新README、项目规划
   - 修改流程 → 更新项目运作流程

3. **技术栈变更后**
   - 更换依赖 → 更新README
   - 更换服务 → 更新README、项目运作流程

4. **审计相关**
   - 发现违规 → 记录到审计违规清单
   - 处理完成 → 更新处理结果

---

## 执行步骤

### 步骤1：识别变更类型

根据用户操作判断需要更新哪些文档：
- 文件结构变化 → 项目地图
- 功能/技术变化 → README、项目规划
- 流程变化 → 项目运作流程
- 违规问题 → 审计违规清单

### 步骤2：读取相关文档

读取需要更新的文档，了解当前内容。

### 步骤3：执行更新

按照以下规则更新文档：

#### 02_项目地图.md 更新规则

```markdown
## 📚 根目录项目文档

新增/删除/修改文件记录

## 📁 src/ 源代码文件

更新文件行数、拆分状态

## 📦 归档区

记录删除的文件及其原因
```

#### 01_README.md 更新规则

- 技术栈变化：更新"技术栈"章节
- 功能变化：更新"功能特性"章节
- 部署变化：更新"部署方式"章节

#### 05_审计违规清单.md 更新规则

```markdown
| 文件 | 问题 | 处理结果 |
|------|------|----------|
| xxx | 问题描述 | 处理方式 |
```

### 步骤4：验证引用

检查文档间的引用是否正确：
- README引用项目地图
- 开发协议引用项目地图
- 审计清单引用具体文件名

## 示例

### 示例1：创建新文件

用户创建了 `src/components/NewFeature.tsx`

**更新操作：**
1. 在 `02_项目地图.md` 的 `src/ 源代码文件` 表格中添加：
   ```markdown
   | **NewFeature.tsx** | 新功能组件 | 新功能描述（XX行） |
   ```

### 示例2：删除废弃文件

用户删除了 `src/utils/oldUtil.ts`

**更新操作：**
1. 在 `02_项目地图.md` 移除该文件记录
2. 在 `02_项目地图.md` 归档区添加：
   ```markdown
   | oldUtil.ts | src/utils/oldUtil.ts | 功能已废弃 | 2026-03-10 |
   ```

### 示例3：拆分超标文件

用户将 `store/index.ts`（174行）拆分为多个模块

**更新操作：**
1. 更新 `02_项目地图.md`：
   ```markdown
   | **store/index.ts** | 状态存储 | 应用全局状态管理（16行，已拆分优化，含7个模块） |
   ```
2. 更新 `05_审计违规清单.md`：
   ```markdown
   | store/index.ts | 174行 | 超过100行限制 | 拆分为7个模块，现16行 |
   ```

---

## 注意事项

1. **即时更新**：文件操作后立即更新文档，不要批量处理
2. **保持一致**：文档间的引用要保持一致
3. **记录归档**：删除的文件要记录归档原因
4. **行数准确**：更新文件行数信息
