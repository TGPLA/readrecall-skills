# 任务闭环验证

## 核心原则

**任务完成 ≠ 代码写完**

真正的完成需要四个步骤：验证 → 文档 → 提交 → 确认

---

## 四大教训

### 1. 每个任务完成后立即验证

| 错误做法 | 正确做法 |
|----------|----------|
| 批量标记任务完成 | 每完成一个任务立即验证 |
| 只检查 TypeScript 编译 | 运行应用，检查控制台 |
| 假设功能正常 | 实际操作验证功能 |

**验证清单**：
```
□ npm run build 构建成功
□ npm run dev 启动无报错
□ 浏览器打开，控制台无错误
□ 功能实际可用（点击、输入、提交）
```

### 2. 文档描述当前实际状态

| 错误做法 | 正确做法 |
|----------|----------|
| 文档描述理想状态 | 文档描述实际状态 |
| 计划标记完成但未执行 | 执行后再标记 |
| 文档与代码不同步 | 变更时同步更新 |

**文档更新规则**：
- 代码变更 → 立即更新相关文档
- 任务状态 → 验证通过后再标记
- 目录结构 → 与实际文件保持一致

### 3. 新组件创建后立即引用验证

| 错误做法 | 正确做法 |
|----------|----------|
| 创建组件但不引用 | 创建后立即在页面中引用 |
| 假设组件会被使用 | 实际调用验证组件可用 |
| 组件成为"孤儿代码" | 确保有入口可访问 |

**引用验证步骤**：
```
1. 创建组件 → 2. 在某页面 import → 3. 在 JSX 中使用 → 4. 验证渲染正常
```

### 4. 开发完成后立即提交

| 错误做法 | 正确做法 |
|----------|----------|
| 文件处于 untracked 状态 | 立即 git add |
| 工作区大量未提交文件 | 每个功能点完成后提交 |
| 代码可能丢失 | 代码安全保存在 Git |

**提交时机**：
- 功能开发完成 → 立即提交
- Bug 修复完成 → 立即提交
- 文档更新完成 → 立即提交

## 闭环验证流程

```
┌─────────────────────────────────────────────────────────────┐
│                      任务闭环验证流程                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 编码                                                    │
│     └── 完成代码编写                                         │
│                                                             │
│  2. 验证 ← 必须实际运行                                      │
│     ├── npm run build                                       │
│     ├── npm run dev                                         │
│     └── 浏览器功能测试                                       │
│                                                             │
│  3. 文档 ← 必须同步更新                                      │
│     ├── 更新项目地图（如有新文件）                            │
│     ├── 更新 README（如有架构变更）                           │
│     └── 更新任务状态（验证后标记）                            │
│                                                             │
│  4. 提交 ← 必须立即提交                                      │
│     ├── git add .                                           │
│     └── git commit -m "..."                                 │
│                                                             │
│  5. 确认                                                    │
│     └── git status 确认工作区干净                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 检查命令

```powershell
# 验证构建
npm run build

# 验证开发服务器
npm run dev

# 检查未跟踪文件
git status --short

# 检查是否有 untracked 文件
git ls-files --others --exclude-standard

# 提交代码
git add .; git commit -m "描述"
```

- 重构代码后

- 修复 Bug 后

## 失败案例参考

**本次功能消失问题的根因**：

1. 提示词组件创建后未引用 → 成为孤儿代码
2. 重构计划标记完成但未验证 → 实际任务未执行
3. 文档描述理想结构 → 与实际代码不一致
4. 文件未提交 Git → 处于 untracked 状态

**教训**：每个环节都缺少验证，导致问题累积。
