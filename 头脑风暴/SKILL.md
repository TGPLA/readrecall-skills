# 头脑风暴

在动手实现之前，先理清**做什么**和**怎么做**。

---

## 何时使用

**需要头脑风暴：**
- 需求不清晰或模糊
- 有多种方案可以解决问题
- 需要和用户讨论权衡
- 用户没有完全表达清楚想要什么
- 功能范围需要明确

**可以直接实现：**
- 需求非常明确具体
- 用户知道确切要什么
- 是简单的 bug 修复或明确定义的变更

---

## 核心流程

### 阶段 0：评估需求清晰度

**需求清晰的信号：**
- 用户提供了具体的验收标准
- 用户引用了现有模式作为参考
- 用户描述了预期的确切行为
- 范围受限且定义明确

**需要头脑风暴的信号：**
- 用户使用了模糊词汇（"优化一下"、"加个类似的功能"）
- 存在多种合理的解释
- 尚未讨论权衡
- 用户似乎对方案不确定

如果需求清晰，建议："你的需求看起来很清晰，可以直接进入规划或实现。"

### 阶段 1：理解想法

一次问一个问题，避免一次问太多。

**提问技巧：**

1. **优先给选项（当有自然选项时）**
   - 好："通知应该是：(a) 仅邮件 (b) 仅应用内 (c) 两者都有？"
   - 差："用户怎么收到通知？"

2. **先广后窄**
   - 首先：核心目的是什么？
   - 然后：用户是谁？
   - 最后：有什么限制？

3. **明确验证假设**
   - "我假设用户已登录，对吗？"

4. **尽早询问成功标准**
   - "你怎么判断这个功能运行良好？"

**关键探索话题：**

| 话题 | 示例问题 |
|------|----------|
| 目的 | 解决什么问题？动机是什么？ |
| 用户 | 谁使用？使用场景是什么？ |
| 限制 | 技术限制？时间线？依赖？ |
| 成功 | 如何衡量成功？理想路径是什么？ |
| 边界情况 | 什么不应该发生？错误状态？ |
| 现有模式 | 代码库中有类似功能吗？ |

**退出条件：** 想法清晰后继续，或用户说"继续"或"我们继续"

### 阶段 2：探索方案

理解想法后，提出 2-3 个具体方案。

**每个方案的结构：**

```markdown
### 方案 A：[名称]

[2-3 句话描述]

**优点：**
- [好处 1]
- [好处 2]

**缺点：**
- [缺点 1]
- [缺点 2]

**最佳场景：** [这个方案最适用的情况]
```

**指南：**
- 优先推荐并解释原因
- 诚实说明权衡
- 考虑 YAGNI——简单通常更好
- 参考代码库现有模式

### 阶段 3：记录设计

用结构化格式总结关键决策。

**设计文档结构：**

```markdown
---
date: YYYY-MM-DD
topic: <kebab-case-topic>
---

# <主题标题>

## 我们要做什么
[简洁描述——最多 1-2 段]

## 为什么选择这个方案
[简要说明考虑的方案及选择原因]

## 关键决策
- [决策 1]：[理由]
- [决策 2]：[理由]

## 未解决问题
- [规划阶段需要解决的任何问题]

## 下一步
→ 进入规划阶段
```

**输出位置：** `docs/brainstorms/YYYY-MM-DD-<topic>-brainstorm.md`

### 阶段 4：交接

提供清晰的选项：

1. **进入规划** → 开始详细规划实现
2. **继续细化** → 继续探索设计
3. **暂存** → 用户稍后返回

---

## YAGNI 原则

积极抵制复杂性：

- **不要为假设的未来需求设计**
- **选择解决陈述问题最简单的方案**
- **优先使用无聊但成熟的模式，而非聪明的方案**
- **当复杂性出现时问"真的需要这个吗？"**
- **推迟不需要现在做的决策**

---

## 增量验证

保持各部分简短——最多 200-300 字。每部分输出后暂停验证理解：

- "这符合你的预期吗？"
- "继续之前有什么需要调整的吗？"
- "这是你想的方向吗？"

---

## 需要避免的反模式

| 反模式 | 更好的做法 |
|--------|------------|
| 一次问 5 个问题 | 一次问一个问题 |
| 直接跳到实现细节 | 聚焦于做什么，不是怎么做 |
| 提出过于复杂的方案 | 从简单开始，只有需要时才增加复杂性 |
| 忽略现有代码库模式 | 先研究已有什么 |
| 不验证就做假设 | 明确说明假设并确认 |
| 创建冗长的设计文档 | 保持简洁——细节放在规划中 |

---

## 与规划的衔接

头脑风暴回答**做什么**：
- 需求和验收标准
- 选择的方案和理由
- 关键决策和权衡

规划回答**怎么做**：
- 实现步骤和文件变更
- 技术细节和代码模式
- 测试策略和验证
